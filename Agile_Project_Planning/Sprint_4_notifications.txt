# Sprint 4 — SMS & Email Notification Integration

**Duration**: Week 8 (5 working days — focused sprint)
**Goal**: Every generated control number automatically triggers an SMS to the
vehicle owner and an email where available — reliably and within seconds.

---

## Sprint Goal
> "The moment a control number is issued, the vehicle owner receives an SMS
> with their bill details within 10 seconds."

---

## User Stories

### US-401 | SMS Notification via Africa's Talking
**As a** vehicle owner,
**I want** to receive an SMS with my parking bill details immediately,
**So that** I know how much to pay and by when.

**Acceptance Criteria:**
- [ ] SMS sent within 10 seconds of control number generation
- [ ] SMS content includes: Control Number, Amount (TZS), Payment Deadline,
      Parking Location, contact info for support
- [ ] SMS is sent to the owner's registered phone number
- [ ] `sms_sent = True` set on ControlNumber record after successful dispatch
- [ ] Failed SMS attempts are logged and retried up to 3 times (via Celery)
- [ ] SMS sending does NOT block the API response (async via Celery)

**SMS Template (English + Swahili):**
```
PARKIPAY - Nambari ya Kudhibiti / Control Number
━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Nambari: PKP-20260513-A3K9P2
Kiasi: TZS 2,000
Muda wa Malipo: leo 14:30
Eneo: Parking Posta, Dar es Salaam
━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Lipa kabla ya muda kumalizika.
Msaada: 0800 110 001
```

**Tasks:**
- [ ] Install `africastalking` Python SDK
- [ ] Create `notifications` Django app
- [ ] Write `SMSService` class:
      — `send_parking_bill_sms(phone, control_number, amount, expires_at, location)`
      — Formats phone to E.164 (+255...)
      — Calls Africa's Talking API
      — Returns `{ success: bool, message_id: str }`
- [ ] Write Celery task `send_sms_notification(control_number_id)`
- [ ] Task retries 3x with exponential backoff on failure
- [ ] Update `ControlNumber.sms_sent = True` on success
- [ ] Log all SMS attempts (status, message_id) to `SMSLog` model
- [ ] Add `AT_API_KEY`, `AT_USERNAME`, `AT_SENDER_ID` to `.env.example`
- [ ] Integration test with Africa's Talking sandbox

### US-402 | Email Notification
**As a** vehicle owner with a registered email,
**I want** to receive a formatted email receipt for my parking bill,
**So that** I have a digital record of the transaction.

**Acceptance Criteria:**
- [ ] Email sent only if `owner_email` is not null/empty
- [ ] Email sent asynchronously (does not delay API response)
- [ ] Email includes all bill details in a clean HTML template
- [ ] Subject: `[ParkiPay] Ankara ya Maegesho / Parking Bill - PKP-XXXXXXXX`
- [ ] `email_sent = True` set on ControlNumber after successful dispatch
- [ ] Failed emails logged; no retry (non-critical path)

**Email HTML Template Includes:**
- ParkiPay logo / header
- Control Number (highlighted)
- Vehicle Plate Number
- Amount Due in TZS
- Payment Deadline
- Parking Location
- Officer Reference (anonymous ID)
- Footer with support contact

**Tasks:**
- [ ] Configure Django email backend: SendGrid (production) / Console (local)
- [ ] Create `email_templates/parking_bill.html` Django template
- [ ] Write `EmailService` class with `send_parking_bill_email()`
- [ ] Write Celery task `send_email_notification(control_number_id)`
- [ ] Update `ControlNumber.email_sent = True` on success
- [ ] Add `SENDGRID_API_KEY`, `FROM_EMAIL` to `.env.example`
- [ ] Test with real email in staging using SendGrid sandbox

---

### US-403 | Notification Status in Bill Success Screen
**As a** field officer,
**I want** to see whether the SMS was sent successfully to the owner,
**So that** I know the owner has been notified and I can proceed.

**Acceptance Criteria:**
- [ ] Bill Success screen shows real-time SMS status: "Sent ✓" or "Pending..."
- [ ] If SMS is delayed (async), screen polls every 3 seconds for up to 15s
- [ ] After 15s if still pending, shows: "SMS being processed — owner will be
      notified shortly"
- [ ] Email status shown only if owner has email (green checkmark or N/A)

**Tasks:**
- [ ] Add `GET /api/billing/{control_number}/status/` endpoint
      Returns: `{ sms_sent, email_sent, status }`
- [ ] On Bill Success screen: poll this endpoint every 3s (max 5 times)
- [ ] Update SMS badge in real time: "Sending..." → "Sent ✓"
- [ ] Stop polling after 15 seconds regardless of status
- [ ] Add to Postman collection

---

### US-404 | Notification Failure Handling
**As a** system administrator,
**I want** all failed notification attempts logged,
**So that** I can diagnose delivery issues and manually follow up if needed.

**Acceptance Criteria:**
- [ ] `SMSLog` model records: phone, control_number_id, attempt_number,
      status [SUCCESS|FAILED|PENDING], provider_message_id, error_reason, timestamp
- [ ] Failed SMS retried 3 times, then marked FAILED permanently
- [ ] Admin can view failed SMS logs in Django Admin
- [ ] Alert (print to Sentry) when SMS failure rate exceeds 10% in an hour

**Tasks:**
- [ ] Create `SMSLog` model + migration
- [ ] Log every SMS attempt (success + failure) to `SMSLog`
- [ ] Register `SMSLog` in Django Admin with filters by status and date
- [ ] Write Celery task failure handler that logs to Sentry
- [ ] Write management command `resend_failed_sms --date=YYYY-MM-DD`
      for manual batch retry by admins

## Notification Flow Diagram

  Officer taps "Generate Bill"
         │
         ▼
  POST /api/billing/generate/
         │
         ├── ControlNumber saved to DB ──────────────────────────────► HTTP 201
         │
         └── Celery task queued (async)
                    │
                    ├── send_sms_notification()
                    │        ├── Africa's Talking API
                    │        ├── Log to SMSLog
                    │        └── Update ControlNumber.sms_sent = True
                    │
                    └── send_email_notification() [if email exists]
                             ├── SendGrid API
                             └── Update ControlNumber.email_sent = True

## Notes & Risks

| Risk                                      | Mitigation                                      |
|-------------------------------------------|-------------------------------------------------|
| Africa's Talking sandbox differs from prod | Test with prod credentials before launch (S6)  |
| Owner phone number format issues           | Normalize to +255 E.164 before sending          |
| High SMS cost at scale                     | Monitor volume; consider bulk SMS agreements    |
| Celery worker down → SMS not sent          | Health check on Celery; manual resend command   |
