# Sprint 1 — Officer Authentication & User Management

**Duration**: Week 2–3 (10 working days)
**Goal**: Field officers can securely log in to the mobile app using government
credentials and stay authenticated during their shift.

---

## Sprint Goal
> "A registered field officer can log in, view their profile, and be logged out
> securely — with every action tied to their verified identity."

---

## User Stories

### US-101 | Officer Login (Backend)
**As a** field officer,
**I want** to log in with my employee ID and password,
**So that** I can access the app securely.

**Acceptance Criteria:**
- [ ] `POST /api/auth/login/` accepts `{ employee_id, password }`
- [ ] Returns `{ access_token, refresh_token, officer_profile }`
- [ ] Invalid credentials return HTTP 401 with clear error message
- [ ] Account locked after 5 consecutive failed attempts (15-min lockout)
- [ ] All login attempts (success + failure) written to `audit_log`

**Tasks:**
- [ ] Create `Officer` model (extends AbstractBaseUser)
      Fields: `employee_id`, `full_name`, `phone`, `email`, `role`,
              `location`, `is_active`, `failed_login_attempts`, `locked_until`
- [ ] Custom authentication backend using `employee_id`
- [ ] Configure SimpleJWT: access token 60min, refresh 7 days
- [ ] Write `LoginView` with lockout logic
- [ ] Write audit log signal on login attempt
- [ ] Write unit tests: valid login, wrong password, locked account

---

### US-102 | Token Refresh
**As a** field officer,
**I want** my session to stay active during a full shift (8–10 hours),
**So that** I am not forced to re-login repeatedly while working.

**Acceptance Criteria:**
- [ ] `POST /api/auth/refresh/` accepts refresh token, returns new access token
- [ ] Expired access tokens return HTTP 401 with `"token_expired"` code
- [ ] Mobile app automatically refreshes token silently before expiry
- [ ] Logout invalidates the refresh token (blacklisting)

**Tasks:**
- [ ] Enable `ROTATE_REFRESH_TOKENS = True` in SimpleJWT settings
- [ ] Enable `BLACKLIST_AFTER_ROTATION = True`
- [ ] Add `rest_framework_simplejwt.token_blacklist` to INSTALLED_APPS
- [ ] Write `LogoutView` that blacklists refresh token
- [ ] On frontend: Axios interceptor to auto-refresh token on 401 response
- [ ] Store tokens securely using `expo-secure-store` (NOT AsyncStorage)

---

### US-103 | Officer Profile Screen (Mobile)
**As a** field officer,
**I want** to see my name, ID, assigned location, and today's date on my
profile screen,
**So that** I can confirm I am logged in as the correct officer.

**Acceptance Criteria:**
- [ ] `GET /api/auth/me/` returns officer profile
- [ ] Profile screen displays: name, employee ID, location, role, today's date
- [ ] "Logout" button logs out and returns to login screen
- [ ] Profile screen is accessible from the app's navigation drawer/tab bar

**Tasks:**
- [ ] `GET /api/auth/me/` endpoint returning officer profile JSON
- [ ] Create `ProfileScreen` in Expo Router (`app/(app)/profile.tsx`)
- [ ] Connect to API using React Query (`useQuery`)
- [ ] Add Zustand `authStore` with `{ officer, token, setAuth, clearAuth }`
- [ ] Implement persistent login using SecureStore (token survives app restart)

---

### US-104 | Role-Based Access Control
**As an** administrator,
**I want** roles enforced at the API level,
**So that** a field officer cannot access admin-only endpoints.

**Acceptance Criteria:**
- [ ] Three roles defined: `FIELD_OFFICER`, `SUPERVISOR`, `ADMIN`
- [ ] Custom DRF permission classes: `IsFieldOfficer`, `IsSupervisor`, `IsAdmin`
- [ ] Admin-only endpoints return HTTP 403 for lower roles
- [ ] Role stored in JWT payload for fast permission checks

**Tasks:**
- [ ] Add `role` field to Officer model with choices
- [ ] Create `permissions.py` with custom DRF permission classes
- [ ] Add `role` claim to JWT via custom token serializer
- [ ] Apply permissions to all relevant views (scaffold for future sprints)
- [ ] Write tests for role enforcement on a sample protected endpoint

---

### US-105 | Login Screen (Mobile)
**As a** field officer,
**I want** a clean, simple login screen on app launch,
**So that** I can enter my credentials quickly before starting my shift.

**Acceptance Criteria:**
- [ ] Login screen shows ParkiPay logo, "Employee ID" field, "Password" field
- [ ] Password field has show/hide toggle
- [ ] Error message displayed below form on invalid credentials
- [ ] Loading indicator shown while API call is in progress
- [ ] On success, navigate to Home screen (plate verification)
- [ ] "Forgot Password" link visible (placeholder for Phase 2)

**Tasks:**
- [ ] Create `app/(auth)/login.tsx` screen
- [ ] Build `LoginForm` component with controlled inputs
- [ ] `useLogin` mutation hook (React Query `useMutation`)
- [ ] Error state handling with user-friendly Swahili + English messages
- [ ] Splash screen with ParkiPay logo (Expo Splash Screen)
- [ ] Navigation guard: redirect to login if no valid token in SecureStore

## API Endpoints — Sprint 1

| Method | Endpoint              | Auth Required | Description              |
|--------|-----------------------|---------------|--------------------------|
| POST   | /api/auth/login/      | No            | Officer login            |
| POST   | /api/auth/refresh/    | No            | Refresh access token     |
| POST   | /api/auth/logout/     | Yes           | Logout + blacklist token |
| GET    | /api/auth/me/         | Yes           | Get officer profile      |

---

## Notes & Risks

| Risk                                        | Mitigation                                    |
|---------------------------------------------|-----------------------------------------------|
| Officers forget passwords in the field      | Add supervisor password-reset endpoint (S6)   |
| Token storage security on Android           | Use expo-secure-store (hardware-backed)       |
| Poor connectivity causing login failures    | Retry logic + offline-friendly error messages |
